# [창업에듀] 웹 서비스 아키텍처 및 보안 운영 구조 분석

본 문서는 현재 운영 중인 **Web(3대) - WAS(2대)** 환경의 계층별 역할, 데이터 흐름, 그리고 보안 체계에 대한 이해를 정리한 검토 보고서입니다.

---

## 1. 서버 구성 및 부하 분산 (Load Balancing)
우리 시스템은 **N:M 연결 구조**를 통해 서비스의 가용성과 성능을 최적화하고 있습니다.

### 1.1 계층별 분업화
* **Web Server (3대):** * **정적 리소스 처리:** HTML, CSS, JS, 이미지 등 변하지 않는 파일을 자체 디스크에서 직접 응답하여 WAS의 부하를 줄임.
    * **리버스 프록시:** 외부 요청을 받아 뒷단의 WAS로 전달하며, SSL 복호화(HTTPS)를 전담함.
* **WAS (2대):** * **동적 로직 처리:** Java/Spring 기반의 비즈니스 로직 수행 및 DB 연동.
    * **보안 필터링:** 유입되는 데이터에 대한 최종 검증 수행.

### 1.2 다중화의 목적
* **고가용성(HA):** 특정 서버 장애 시에도 나머지 서버가 서비스를 지속(Failover).
* **확장성:** 트래픽 증가 시 Web 또는 WAS를 독립적으로 증설 가능.

---

## 2. 데이터 저장 및 공유 구조 (Storage)
여러 대의 WAS가 동일한 데이터를 처리하기 위해 **공통 저장소**를 활용합니다.

### 2.1 NAS (Network Attached Storage) - "공용 창고"
* **역할:** 사용자 업로드 파일(이미지, 첨부파일 등)을 물리적으로 저장.
* **공유 원리:** WAS 1번과 2번이 동일한 NAS 경로를 마운트하여 사용하므로, 어떤 서버를 통해서 접속하더라도 동일한 파일에 접근 가능.

### 2.2 DB Server - "데이터 장부"
* **역할:** 사용자 정보, 게시글 텍스트 등 정형화된 데이터를 저장.
* **연결 관계:** WAS는 DB와 직접 통신하며, DB에는 **NAS에 저장된 실제 파일의 경로(Path)**를 기록함.

> **주의:** WAS는 NAS와 DB를 각각 별도의 통로로 연결하며, NAS와 DB는 서로 직접 통신하지 않음.

---

## 3. 보안 솔루션 및 필터링 (Security)
안전한 데이터 유입을 위해 다중 필터링 체계를 구축하고 있습니다.

| 솔루션 | 주요 역할 | 비유 |
| :--- | :--- | :--- |
| **WebFilter** | 웹 해킹 공격(SQL Injection, XSS 등) 차단 | 외부 침입 방지 파수꾼 |
| **EZFilter** | 개인정보(주민번호, 연락처 등) 유출 방지 및 마스킹 | 내부 정보 유출 검문소 |

---

## 4. 네트워크 구간 및 방화벽 (DMZ)
망 분리 원칙에 따라 구간별 접근 권한을 엄격히 통제합니다.

1.  **외부 → DMZ (Web Server):** * **443(HTTPS) 포트**를 `Any`(전체) 대상으로 개방하여 외부 접근성 확보.
    * 이 구간에서 SSL 인증서를 통한 구간 암호화 적용.
2.  **DMZ (Web) → 내부망 (WAS):** * `Any` 접속 절대 불가. 오직 **웹 서버 3대의 IP**만 WAS 전용 포트로 접속 허용.
3.  **내부망 (WAS) → DB/NAS:** * 가장 깊은 내부망으로, 인가된 WAS 서버만 접근 가능하도록 2차 방화벽 구성.

---

## 5. 결론: 시스템 운영 핵심 요약
* **독립성:** 웹 서버는 자체 리소스로 정적 파일을 빠르게 서비스함.
* **일관성:** NAS와 DB를 통해 다중 WAS 환경에서도 데이터의 동일성을 보장함.
* **안전성:** 외부에는 최소한의 문(443)만 열되, 내부로 갈수록 IP 기반의 화이트리스트 방식으로 방어함.
